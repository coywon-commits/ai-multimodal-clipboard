# 출시 전략 문서 (GitHub → 익스텐션 전용 → 웹스토어)

## 한 줄 요약
오픈소스 신뢰도와 사용자 유입은 **GitHub(파이썬 포함 풀버전)** 으로 먼저 만들고, **익스텐션 전용(크롬에서만 복붙용)** 을 안정화한 뒤 **Chrome 웹스토어** 로 확장한다.

## 왜 이 순서가 좋은가
1. **GitHub** 는 개발자/초기 사용자에게 빠르게 공개 가능하고, 설치/버그 제보/기여 흐름을 일찍 만들 수 있음.
2. **익스텐션 전용** 으로 일반 사용자들이 “복붙이 된다는” 핵심 경험을 더 단순하게 제공 가능.
3. **웹스토어** 는 심사/정리(권한, 스크린샷, 개인정보 처리 등)가 필요하지만, 출시 후 유저 유입이 폭발적으로 증가할 수 있음.

## 릴리즈 구성 (3단계)

### 1단계. GitHub 공개 (Python + Chrome 확장 풀버전)
**목표:** 지금 당장 “동작하는 가치”를 보여주고, 초기 피드백을 확보한다.

**포함 구성**
- `chrome-extension/` : AI 채팅에서 텍스트/이미지 추출
- `python-app/` : 로컬 저장 데이터를 다른 AI 채팅에 자동 붙여넣기

**사용자 가치(설명 포인트)**
- 텍스트는 물론 이미지까지 한 번에 다루며, “다운로드 후 재업로드” 불편을 줄임.
- 단축키 기반 자동 붙여넣기 제공 (`Ctrl+Shift+V`)

**README에 반드시 넣을 것(최소 세트)**
- 빠른 시작(설치 순서가 한눈에 보이게)
- 지원 사이트 목록(현재/예정)
- 권한/제한사항(CORS/사이트 구조 변화 가능성 등)
- 트러블슈팅(이미지 추출/붙여넣기 안 될 때)
- 문제 제보/기능 요청(이슈 템플릿 또는 가이드 링크)

**운영 포인트**
- 초기 릴리즈 버전은 “MVP”로 명확히 표기(기대치 관리).
- “알려진 이슈” 섹션을 최우선으로 운영.

---

### 2단계. 익스텐션 전용 버전 출시 계획 (Chrome에서만 복붙용)
**목표:** “각 AI 창 사이에서 복붙”을 **파이썬 없이도** 사용 가능하게 만들어, 일반 유저 진입장벽을 낮춘다.

**정의(이 문서에서의 의미)**
- 사용자 기기에서 별도 Python 앱 설치가 필요 없고,
- 사용 경험은 Chrome 확장 UI(버튼/팝업) 중심으로 제공한다.
- 붙여넣기는 “확장만으로 지원 가능한 범위”에서 구현하고, 불가능한 부분이 있다면 README에 투명하게 안내한다.

**포함 구성(권장)**
- `chrome-extension/` 단독 배포
- 로컬 저장(또는 확장 내부 상태) + 클립보드/DOM 삽입/업로드 입력 트리거 등
- “복사 → 대상 AI 창에서 붙여넣기” 워크플로우를 팝업 또는 버튼으로 제공

**출시 패키징(권장 선택지)**
- 선택지 A: GitHub 내 `extension-only` 브랜치/폴더 구조로 관리
- 선택지 B: Releases에 “Extension Only” 빌드/소스를 별도 자산으로 제공

**README에 반드시 넣을 것(최소 세트)**
- “Python 앱이 필요 없다”를 최상단에 명시
- 사용 흐름(예: ① 원본 AI에서 추출 버튼 ② 대상 AI로 이동 ③ 붙여넣기 버튼)
- 지원 사이트 범위(2단계에서 확실한 사이트부터)
- 제한사항(확장만으로 어려운 이미지 처리/업로드 제약이 있다면 공개)

**운영 포인트**
- 웹스토어 심사 준비를 염두에 두고, 권한/설명/스크린샷을 2단계에서 미리 정리.
- 사용자 피드백을 “웹스토어용 전용 기능”에 직접 반영.

---

### 3단계. Chrome 웹스토어 출시
**목표:** 심사 통과 후 클릭 한 번 설치로 유저 유입을 극대화한다.

**예상 일정**
- 심사: 통상 2~3일 범위 내에서 계획
- 2단계 안정화 후, 웹스토어용 소재/문구를 정리해 제출

**웹스토어에 준비해야 할 것(체크리스트)**
- 확장 스토어용 스크린샷(사용 흐름이 보이게: 추출 버튼/붙여넣기 버튼/결과)
- 권한(Permissions) 최소화 및 이유 설명
- 개인정보 처리/쿠키/데이터 수집 여부 관련 문구(필요 시 별도 링크)
- 지원/문의 채널(이메일 또는 GitHub 링크)
- 버전/변경사항(Release notes) 명확화

## 일정(추천)
1. **즉시(오늘~하루):** 1단계 GitHub 공개(풀버전) + README 한/영 초안 작성
2. **1~2주:** 2단계 익스텐션 전용 구현/정리 + 웹스토어 소재 초안
3. **심사(2~3일):** 웹스토어 제출 후 대기 + 초기 리뷰 대응 체계 준비

## README 한/영 작성 가이드(짧은 체크리스트)
- 같은 목차로 한국어/영어를 맞추기(사용자 혼란 최소화)
- 상단에 “이 프로젝트가 해결하는 문제 1줄”을 동일하게 유지
- Quick Start는 단계별 번호로 작성
- Known Limitations / Troubleshooting을 영어에서도 동일하게 반영
- 이미지가 핵심 가치라면, 영어 README에도 결과 캡처 1장 이상 권장
